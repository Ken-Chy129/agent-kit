# 规划与任务拆解

## 概述

将工作分解为小的、可验证的任务，附带明确的验收标准。良好的任务拆解是 Agent 可靠完成工作与产出一团乱麻之间的分水岭。每个任务都应足够小，能在一个聚焦的会话中实现、测试和验证。

## 何时使用

- 你有规范，需要将其拆分为可实现的单元
- 任务感觉太大或太模糊而无法开始
- 工作需要在多个 Agent 或会话之间并行化
- 你需要向人类沟通范围
- 实现顺序不明显

**何时不使用：** 范围明显的单文件变更，或规范中已经包含定义良好的任务。

## 规划流程

### 步骤 1：进入规划模式

在编写任何代码之前，以只读模式运作：

- 阅读规范和相关代码库部分
- 识别现有模式和约定
- 映射组件之间的依赖关系
- 记录风险和未知项

**在规划期间不要编写代码。** 输出是一份计划文档，不是实现。

### 步骤 2：识别依赖图

映射什么依赖于什么：

```
数据库 Schema
    │
    ├── API 模型/类型
    │       │
    │       ├── API 端点
    │       │       │
    │       │       └── 前端 API 客户端
    │       │               │
    │       │               └── UI 组件
    │       │
    │       └── 验证逻辑
    │
    └── 种子数据 / 迁移
```

实现顺序遵循依赖图自底向上：先构建基础。

### 步骤 3：垂直切片

不要先构建所有数据库、再构建所有 API、再构建所有 UI——而是每次构建一条完整的功能路径：

**差（水平切片）：**
```
任务 1: 构建整个数据库 Schema
任务 2: 构建所有 API 端点
任务 3: 构建所有 UI 组件
任务 4: 把一切连接起来
```

**好（垂直切片）：**
```
任务 1: 用户可以创建账户 (注册的 Schema + API + UI)
任务 2: 用户可以登录 (认证的 Schema + API + UI)
任务 3: 用户可以创建任务 (任务的 Schema + API + UI)
任务 4: 用户可以查看任务列表 (查询 + API + 列表视图 UI)
```

每个垂直切片交付可工作、可测试的功能。

### 步骤 4：编写任务

每个任务遵循以下结构：

```markdown
## 任务 [N]: [简短描述性标题]

**描述：** 一段话解释这个任务完成什么。

**验收标准：**
- [ ] [具体的、可测试的条件]
- [ ] [具体的、可测试的条件]

**验证：**
- [ ] 测试通过: `npm test -- --grep "feature-name"`
- [ ] 构建成功: `npm run build`
- [ ] 手动检查: [描述要验证什么]

**依赖：** [此任务依赖的任务编号，或"无"]

**可能涉及的文件：**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**预估范围：** [小: 1-2 个文件 | 中: 3-5 个文件 | 大: 5+ 个文件]
```

### 步骤 5：排序与设置检查点

安排任务使得：

1. 依赖关系得到满足（先构建基础）
2. 每个任务都让系统保持在可工作状态
3. 每 2-3 个任务后设置验证检查点
4. 高风险任务排在前面（快速失败）

添加明确的检查点：

```markdown
## 检查点：任务 1-3 完成后
- [ ] 所有测试通过
- [ ] 应用构建无错误
- [ ] 核心用户流程端到端可用
- [ ] 继续之前与人类审查
```

## 任务规模指南

| 规模 | 文件数 | 范围 | 示例 |
|------|-------|------|---------|
| **XS** | 1 | 单个函数或配置变更 | 添加一条验证规则 |
| **S** | 1-2 | 一个组件或端点 | 添加一个新的 API 端点 |
| **M** | 3-5 | 一个功能切片 | 用户注册流程 |
| **L** | 5-8 | 多组件功能 | 带过滤和分页的搜索 |
| **XL** | 8+ | **太大了——需要进一步拆分** | — |

如果任务是 L 或更大，应该拆分为更小的任务。Agent 在 S 和 M 任务上表现最好。

**何时需要进一步拆分任务：**
- 需要超过一个聚焦会话（大约 2+ 小时的 Agent 工作）
- 你无法用 3 个或更少的要点描述验收标准
- 它涉及两个或更多独立的子系统（如认证和计费）
- 你发现自己在任务标题中写了"和"（这表明它是两个任务）

## 计划文档模板

```markdown
# 实施计划: [功能/项目名称]

## 概述
[一段话总结我们要构建什么]

## 架构决策
- [关键决策 1 及理由]
- [关键决策 2 及理由]

## 任务列表

### 阶段 1: 基础
- [ ] 任务 1: ...
- [ ] 任务 2: ...

### 检查点: 基础
- [ ] 测试通过，构建干净

### 阶段 2: 核心功能
- [ ] 任务 3: ...
- [ ] 任务 4: ...

### 检查点: 核心功能
- [ ] 端到端流程可用

### 阶段 3: 完善
- [ ] 任务 5: ...
- [ ] 任务 6: ...

### 检查点: 完成
- [ ] 所有验收标准满足
- [ ] 准备好审查

## 风险与缓解措施
| 风险 | 影响 | 缓解策略 |
|------|--------|------------|
| [风险] | [高/中/低] | [策略] |

## 待解决问题
- [需要人类输入的问题]
```

## 并行化机会

当多个 Agent 或会话可用时：

- **可以安全并行的：** 独立的功能切片、已实现功能的测试、文档
- **必须顺序执行的：** 数据库迁移、共享状态变更、依赖链
- **需要协调的：** 共享 API 契约的功能（先定义契约，然后并行化）

## 常见合理化借口

| 合理化借口 | 现实 |
|---|---|
| "我边做边想" | 这样最终会产生混乱和返工。10 分钟的规划能节省数小时。 |
| "任务很明显" | 还是写下来。明确的任务能暴露隐藏的依赖关系和被遗忘的边界情况。 |
| "规划是开销" | 规划就是任务本身。没有计划的实现只是打字。 |
| "我能全记在脑子里" | 上下文窗口是有限的。书面计划能跨会话和上下文压缩存活。 |

## 危险信号

- 没有书面任务列表就开始实现
- 任务写着"实现该功能"却没有验收标准
- 计划中没有验证步骤
- 所有任务都是 XL 规模
- 任务之间没有检查点
- 没有考虑依赖顺序

## 验证

在开始实现前，确认：

- [ ] 每个任务都有验收标准
- [ ] 每个任务都有验证步骤
- [ ] 任务依赖关系已识别并正确排序
- [ ] 没有任务涉及超过约 5 个文件
- [ ] 主要阶段之间存在检查点
- [ ] 人类已审查并批准了计划
