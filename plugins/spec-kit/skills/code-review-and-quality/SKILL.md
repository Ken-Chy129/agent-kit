# 代码审查与质量

## 概述

多维度代码审查与质量门禁。每个变更在合并前都要经过审查——没有例外。审查涵盖五个维度：正确性、可读性、架构、安全性和性能。

**批准标准：** 当变更确实改善了整体代码健康状况时就批准它，即使它并不完美。完美的代码不存在——目标是持续改进。不要因为代码不完全是你会写的方式而阻止变更。如果它改善了代码库并遵循项目约定，就批准它。

## 何时使用

- 合并任何 PR 或变更之前
- 完成功能实现之后
- 当其他 Agent 或模型生成了需要你评估的代码时
- 重构现有代码时
- 任何 bug 修复之后（同时审查修复和回归测试）

## 五维度审查

每次审查都从以下维度评估代码：

### 1. 正确性

代码是否做了它声称要做的事？

- 是否符合规范或任务要求？
- 是否处理了边界情况（null、空值、边界值）？
- 是否处理了错误路径（不仅仅是正常路径）？
- 是否通过所有测试？测试是否确实在测试正确的东西？
- 是否存在差一错误、竞态条件或状态不一致？

### 2. 可读性与简洁性

其他工程师（或 Agent）能否在没有作者解释的情况下理解这段代码？

- 命名是否描述性强且与项目约定一致？（不要使用没有上下文的 `temp`、`data`、`result`）
- 控制流是否直观（避免嵌套三元运算、深层回调）？
- 代码是否逻辑组织（相关代码分组、清晰的模块边界）？
- 是否有应该简化的"聪明"技巧？
- **能否用更少的代码完成？**（100 行能完成的事写了 1000 行是失败的）
- **抽象是否值得其复杂度？**（在第三个用例出现之前不要泛化）
- 注释是否有助于阐明非显而易见的意图？（但不要给显而易见的代码加注释。）
- 是否有死代码痕迹：无用变量（`_unused`）、向后兼容的垫片、或 `// removed` 注释？

### 3. 架构

变更是否符合系统设计？

- 是否遵循现有模式还是引入了新模式？如果是新模式，是否合理？
- 是否保持了清晰的模块边界？
- 是否存在应该共享的代码重复？
- 依赖方向是否正确（无循环依赖）？
- 抽象层级是否合适（不过度工程化，不过度耦合）？

### 4. 安全性

详细的安全指南请参见 `security-and-hardening`。变更是否引入了漏洞？

- 用户输入是否经过验证和清理？
- 密钥是否远离代码、日志和版本控制？
- 是否在需要的地方检查了认证/授权？
- SQL 查询是否参数化（无字符串拼接）？
- 输出是否编码以防止 XSS？
- 依赖是否来自可信来源且无已知漏洞？
- 来自外部来源（API、日志、用户内容、配置文件）的数据是否被视为不可信？
- 外部数据流在用于逻辑或渲染之前，是否在系统边界处进行了验证？

### 5. 性能

详细的性能分析和优化请参见 `performance-optimization`。变更是否引入了性能问题？

- 是否存在 N+1 查询模式？
- 是否存在无界循环或无约束的数据获取？
- 是否有应该异步的同步操作？
- UI 组件是否存在不必要的重新渲染？
- 列表端点是否缺少分页？
- 是否在热路径中创建了大对象？

## 变更规模控制

小而集中的变更更容易审查、更快合并、更安全部署。目标规模：

```
~100 行变更   → 好。可以一次审查完。
~300 行变更   → 如果是单个逻辑变更则可接受。
~1000 行变更  → 太大了。拆分它。
```

**什么算"一个变更"：** 一个独立的修改，解决一件事，包含相关测试，并在提交后保持系统功能正常。功能的一部分——而非整个功能。

**变更过大时的拆分策略：**

| 策略 | 方法 | 适用场景 |
|------|------|----------|
| **堆叠** | 提交一个小变更，基于它开始下一个 | 有顺序依赖时 |
| **按文件分组** | 将需要不同审查者的文件组分开提交 | 跨领域关注点 |
| **水平拆分** | 先创建共享代码/桩代码，再创建消费者 | 分层架构 |
| **垂直拆分** | 将功能拆分为更小的全栈切片 | 功能开发 |

**大变更可接受的情况：** 完整文件删除和自动化重构，审查者只需验证意图而无需逐行检查。

**将重构与功能开发分开。** 既重构现有代码又添加新行为的变更是两个变更——分别提交。小的清理（变量重命名）可由审查者酌情决定是否包含。

## 变更描述

每个变更都需要在版本控制历史中独立可读的描述。

**第一行：** 简短、祈使语气、独立完整。写 "Delete the FizzBuzz RPC" 而非 "Deleting the FizzBuzz RPC"。必须足够信息丰富，让搜索历史的人无需阅读 diff 就能理解变更。

**正文：** 说明什么在变更以及为什么。包含代码本身无法体现的上下文、决策和推理。在相关时链接到 bug 编号、基准测试结果或设计文档。在存在缺陷时承认方案的不足。

**反模式：** "Fix bug"、"Fix build"、"Add patch"、"Moving code from A to B"、"Phase 1"、"Add convenience functions"。

## 审查流程

### 第一步：理解上下文

在查看代码之前，先理解意图：

```
- 这个变更要完成什么？
- 它实现了什么规范或任务？
- 预期的行为变更是什么？
```

### 第二步：先审查测试

测试揭示意图和覆盖范围：

```
- 变更是否有测试？
- 测试是否测试行为（而非实现细节）？
- 是否覆盖了边界情况？
- 测试是否有描述性名称？
- 如果代码发生变化，测试能否捕获回归？
```

### 第三步：审查实现

带着五个维度逐一审查代码：

```
对每个变更的文件：
1. 正确性：这段代码是否做了测试所说的事？
2. 可读性：我能不借助帮助理解这段代码吗？
3. 架构：这是否符合系统设计？
4. 安全性：是否有漏洞？
5. 性能：是否有瓶颈？
```

### 第四步：分类发现

用严重程度标注每条评论，让作者知道什么是必须的、什么是可选的：

| 前缀 | 含义 | 作者操作 |
|------|------|----------|
| *（无前缀）* | 需要修改 | 合并前必须处理 |
| **Critical:** | 阻止合并 | 安全漏洞、数据丢失、功能损坏 |
| **Nit:** | 次要，可选 | 作者可忽略——格式、风格偏好 |
| **Optional:** / **Consider:** | 建议 | 值得考虑但非必需 |
| **FYI** | 仅供参考 | 无需操作——为未来提供上下文参考 |

这防止了作者将所有反馈都视为强制要求，从而在可选建议上浪费时间。

### 第五步：验证验证过程

检查作者的验证情况：

```
- 运行了什么测试？
- 构建是否通过？
- 是否进行了手动测试？
- UI 变更是否有截图？
- 是否有修改前/后的对比？
```

## 多模型审查模式

使用不同的模型从不同角度进行审查：

```
模型 A 编写代码
    │
    ▼
模型 B 审查正确性和架构
    │
    ▼
模型 A 处理反馈
    │
    ▼
人类做最终决定
```

这能捕获单个模型可能遗漏的问题——不同模型有不同的盲点。

**审查 Agent 的示例提示：**
```
审查这个代码变更的正确性、安全性以及是否符合我们的项目约定。
规范要求 [X]。变更应该 [Y]。
将问题标记为 Critical、Important 或 Suggestion。
```

## 死代码清理

在任何重构或实现变更之后，检查孤立代码：

1. 识别现在不可达或未使用的代码
2. 明确列出它们
3. **删除前先询问：** "是否应该移除这些现在未使用的元素：[列表]？"

不要让死代码留在那里——它会让未来的读者和 Agent 困惑。但也不要默默删除你不确定的东西。有疑问时就问。

```
已识别的死代码：
- src/utils/date.ts 中的 formatLegacyDate() — 已被 formatDate() 替代
- src/components/ 中的 OldTaskCard 组件 — 已被 TaskCard 替代
- src/config.ts 中的 LEGACY_API_URL 常量 — 无剩余引用
→ 可以安全移除这些吗？
```

## 审查速度

缓慢的审查会阻塞整个团队。切换上下文进行审查的成本，低于让他人等待的成本。

- **在一个工作日内响应**——这是上限，不是目标
- **理想节奏：** 在审查请求到达后尽快响应，除非正在深度编码。典型的变更应该在一天内完成多轮审查
- **优先快速的单次响应**而非快速的最终批准。即使需要多轮审查，快速反馈也能减少挫败感
- **大变更：** 要求作者拆分它们，而不是审查一个巨大的变更集

## 处理分歧

解决审查争议时，应用以下优先级：

1. **技术事实和数据**优于观点和偏好
2. **风格指南**是风格问题的绝对权威
3. **软件设计**必须基于工程原则评估，而非个人偏好
4. **代码库一致性**在不降低整体健康度的前提下是可接受的

**不要接受"我以后再清理"。** 经验表明延迟的清理很少发生。要求在提交前清理，除非是真正的紧急情况。如果周边问题无法在本次变更中处理，要求创建 bug 并自行认领。

## 审查中的诚实

审查代码时——无论是你自己、其他 Agent 还是人类编写的：

- **不要橡皮图章。** 没有审查证据的 "LGTM" 对谁都没帮助。
- **不要淡化真实问题。** 当一个 bug 会在生产环境中出现时说"这可能是个小问题"是不诚实的。
- **尽可能量化问题。** "这个 N+1 查询会为列表中每个项目增加约 50ms" 比 "这可能会慢" 更好。
- **对有明显问题的方案直接提出质疑。** 审查中的阿谀奉承是一种失败模式。如果实现有问题，直接说出来并提出替代方案。
- **优雅地接受否决。** 如果作者掌握完整上下文且不同意，尊重他们的判断。评论代码而非评论人——将个人批评转化为对代码本身的关注。

## 依赖纪律

代码审查的一部分是依赖审查：

**添加任何依赖之前：**
1. 现有技术栈能解决吗？（通常可以。）
2. 依赖有多大？（检查包体积影响。）
3. 是否在积极维护？（检查最近提交、未关闭的 issue。）
4. 是否有已知漏洞？（`npm audit`）
5. 许可证是什么？（必须与项目兼容。）

**规则：** 优先使用标准库和现有工具而非新依赖。每个依赖都是一种负债。

## 审查清单

```markdown
## 审查：[PR/变更标题]

### 上下文
- [ ] 我理解这个变更做了什么以及为什么

### 正确性
- [ ] 变更符合规范/任务要求
- [ ] 边界情况已处理
- [ ] 错误路径已处理
- [ ] 测试充分覆盖变更

### 可读性
- [ ] 命名清晰且一致
- [ ] 逻辑直观
- [ ] 无不必要的复杂度

### 架构
- [ ] 遵循现有模式
- [ ] 无不必要的耦合或依赖
- [ ] 抽象层级合适

### 安全性
- [ ] 代码中无密钥
- [ ] 边界处验证了输入
- [ ] 无注入漏洞
- [ ] 认证检查就位
- [ ] 外部数据源被视为不可信

### 性能
- [ ] 无 N+1 模式
- [ ] 无无界操作
- [ ] 列表端点有分页

### 验证
- [ ] 测试通过
- [ ] 构建成功
- [ ] 手动验证完成（如适用）

### 结论
- [ ] **批准** — 可以合并
- [ ] **请求变更** — 必须处理问题
```

## 常见的自我合理化

| 合理化借口 | 现实 |
|---|---|
| "能用就行" | 能用但不可读、不安全或架构有问题的代码会产生复利式增长的技术债务。 |
| "我写的，我知道是对的" | 作者对自己的假设是盲目的。每个变更都受益于另一双眼睛。 |
| "我们以后再清理" | 以后永远不会来。审查就是质量门禁——用好它。要求合并前清理，而非合并后。 |
| "AI 生成的代码应该没问题" | AI 代码需要更多审查，而非更少。它自信且看似合理，即使是错的。 |
| "测试通过了，所以没问题" | 测试是必要条件但非充分条件。它们无法捕获架构问题、安全问题或可读性问题。 |

## 危险信号

- PR 在没有任何审查的情况下合并
- 审查只检查测试是否通过（忽略其他维度）
- 没有实际审查证据的 "LGTM"
- 安全敏感的变更没有安全专项审查
- "太大无法好好审查"的大 PR（拆分它们）
- bug 修复 PR 没有回归测试
- 审查评论没有严重程度标签——不清楚什么是必需的、什么是可选的
- 接受"我以后再修"——永远不会发生

## 验证清单

审查完成后：

- [ ] 所有 Critical 问题已解决
- [ ] 所有 Important 问题已解决或有明确的推迟理由
- [ ] 测试通过
- [ ] 构建成功
- [ ] 验证过程已记录（什么变了，如何验证的）
