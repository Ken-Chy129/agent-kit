# 规格驱动开发

## 概述

在编写任何代码之前，先编写一份结构化的规格说明。规格说明是你和人类工程师之间共享的真实来源——它定义了我们要构建什么、为什么要构建，以及如何判断它已完成。没有规格说明的代码就是在猜测。

## 何时使用

- 启动新项目或新功能
- 需求模棱两可或不完整
- 变更涉及多个文件或模块
- 即将做出架构决策
- 实现需要超过 30 分钟的任务

**不适用场景：** 单行修复、拼写错误修正，或需求明确且自包含的变更。

## 门控工作流

规格驱动开发分为四个阶段。在当前阶段验证通过之前，不要进入下一阶段。

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
（规格）     （计划）  （任务）   （实现）
   │          │        │          │
   ▼          ▼        ▼          ▼
 人工        人工      人工       人工
 审查        审查      审查       审查
```

### 阶段 1：规格

从高层愿景开始。向人类提出澄清问题，直到需求变得具体。

**立即暴露假设。** 在编写任何规格内容之前，列出你正在做的假设：

```
我正在做的假设：
1. 这是一个 Web 应用（非原生移动端）
2. 认证使用基于会话的 cookie（非 JWT）
3. 数据库是 PostgreSQL（基于现有的 Prisma schema）
4. 我们只面向现代浏览器（不支持 IE11）
→ 请现在纠正我，否则我将按这些假设继续。
```

不要默默填补模糊的需求。规格说明的全部意义在于*在代码编写之前*暴露误解——假设是最危险的误解形式。

**编写覆盖以下六个核心领域的规格文档：**

1. **目标** —— 我们在构建什么，为什么？谁是用户？成功是什么样的？

2. **命令** —— 带有参数的完整可执行命令，而不仅仅是工具名称。
   ```
   构建：npm run build
   测试：npm test -- --coverage
   代码检查：npm run lint --fix
   开发：npm run dev
   ```

3. **项目结构** —— 源代码、测试和文档的存放位置。
   ```
   src/           → 应用源代码
   src/components → React 组件
   src/lib        → 共享工具
   tests/         → 单元测试和集成测试
   e2e/           → 端到端测试
   docs/          → 文档
   ```

4. **代码风格** —— 一段展示你的代码风格的真实代码片段比三段描述性文字更有效。包括命名约定、格式规则和良好输出的示例。

5. **测试策略** —— 使用什么框架、测试存放在哪里、覆盖率期望、哪些测试级别对应哪些关注点。

6. **边界** —— 三级体系：
   - **始终执行：** 提交前运行测试、遵循命名约定、验证输入
   - **先询问：** 数据库 schema 变更、添加依赖、更改 CI 配置
   - **绝不执行：** 提交密钥、编辑 vendor 目录、未经批准删除失败的测试

**规格模板：**

```markdown
# 规格：[项目/功能名称]

## 目标
[我们在构建什么以及为什么。用户故事或验收标准。]

## 技术栈
[框架、语言、关键依赖及版本]

## 命令
[构建、测试、代码检查、开发——完整命令]

## 项目结构
[目录布局及描述]

## 代码风格
[示例代码片段 + 关键约定]

## 测试策略
[框架、测试位置、覆盖率要求、测试级别]

## 边界
- 始终：[...]
- 先询问：[...]
- 绝不：[...]

## 成功标准
[如何判断这项工作已完成——具体的、可测试的条件]

## 待解决问题
[任何未解决的、需要人工输入的问题]
```

**将指令重新表述为成功标准。** 当收到模糊需求时，将其转化为具体条件：

```
需求："让仪表板更快"

重新表述的成功标准：
- 仪表板 LCP 在 4G 网络下 < 2.5s
- 初始数据加载在 < 500ms 内完成
- 加载过程中无布局偏移（CLS < 0.1）
→ 这些是正确的目标吗？
```

这让你可以朝着明确的目标进行循环、重试和解决问题，而不是猜测"更快"意味着什么。

### 阶段 2：计划

根据已验证的规格，生成技术实现计划：

1. 识别主要组件及其依赖关系
2. 确定实现顺序（什么必须先构建）
3. 记录风险和缓解策略
4. 识别哪些可以并行构建，哪些必须串行
5. 定义阶段间的验证检查点

计划应该是可审查的：人类应该能够阅读后说"是的，这是正确的方法"或"不，请改变 X"。

### 阶段 3：任务

将计划分解为离散的、可实现的任务：

- 每个任务应该能在一次专注的工作会话中完成
- 每个任务有明确的验收标准
- 每个任务包含验证步骤（测试、构建、手动检查）
- 任务按依赖关系排序，而非按重要性感知
- 每个任务涉及的文件不应超过约 5 个

**任务模板：**
```markdown
- [ ] 任务：[描述]
  - 验收：[完成时必须为真的条件]
  - 验证：[如何确认——测试命令、构建、手动检查]
  - 文件：[将涉及哪些文件]
```

### 阶段 4：实现

按照 `incremental-implementation`（增量实现）和 `test-driven-development`（测试驱动开发）技能逐个执行任务。使用 `context-engineering`（上下文工程）在每一步加载正确的规格章节和源文件，而不是将整个规格灌入 Agent。

## 保持规格的活力

规格是一份活文档，不是一次性产物：

- **决策变更时更新** —— 如果发现数据模型需要变更，先更新规格，再实现。
- **范围变更时更新** —— 新增或删减的功能应反映在规格中。
- **提交规格** —— 规格应与代码一起存放在版本控制中。
- **在 PR 中引用规格** —— 将每个 PR 链接回其所实现的规格章节。

## 常见自我合理化

| 合理化借口 | 现实 |
|---|---|
| "这很简单，不需要规格" | 简单任务不需要*长篇*规格，但仍需要验收标准。两行的规格也可以。 |
| "我写完代码再写规格" | 那是文档，不是规格说明。规格的价值在于*编码之前*迫使你理清思路。 |
| "规格会拖慢我们" | 15 分钟的规格能防止数小时的返工。15 分钟的瀑布式规格好过 15 小时的调试。 |
| "反正需求会变" | 这正是规格作为活文档的原因。过时的规格仍然比没有规格好。 |
| "用户知道他们想要什么" | 即使是清晰的需求也有隐含假设。规格就是让这些假设浮出水面。 |

## 危险信号

- 没有任何书面需求就开始写代码
- 在澄清"完成"意味着什么之前就问"我直接开始做吗？"
- 实现规格或任务列表中未提及的功能
- 做出架构决策却不记录
- 因为"要做什么很明显"就跳过规格

## 验证

在进入实现之前，确认：

- [ ] 规格覆盖了所有六个核心领域
- [ ] 人类已审查并批准了规格
- [ ] 成功标准是具体的、可测试的
- [ ] 边界（始终/先询问/绝不）已定义
- [ ] 规格已保存为仓库中的文件
