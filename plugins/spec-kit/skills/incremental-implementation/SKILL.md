# 增量式实现

## 概述

以薄的垂直切片进行构建——实现一个部分，测试它，验证它，然后扩展。避免一次性实现整个功能。每个增量都应让系统保持在可工作、可测试的状态。这是使大型功能变得可管理的执行纪律。

## 何时使用

- 实现任何多文件变更
- 从任务分解构建新功能
- 重构现有代码
- 任何你想在测试前编写超过约 100 行代码的时候

**何时不使用：** 范围已经很小的单文件、单函数变更。

## 增量循环

```
┌──────────────────────────────────────┐
│                                      │
│   实现 ──→ 测试 ──→ 验证 ──┐        │
│       ▲                     │        │
│       └───── 提交 ◄─────────┘        │
│              │                       │
│              ▼                       │
│          下一个切片                   │
│                                      │
└──────────────────────────────────────┘
```

对每个切片：

1. **实现** 最小的完整功能片段
2. **测试** —— 运行测试套件（如果没有测试则编写一个）
3. **验证** —— 确认切片按预期工作（测试通过、构建成功、手动检查）
4. **提交** —— 用描述性消息保存你的进度（参见 `git-workflow-and-versioning` 的原子提交指导）
5. **进入下一个切片** —— 继续前进，不要重来

## 切片策略

### 垂直切片（推荐）

构建贯穿技术栈的一条完整路径：

```
切片 1: 创建任务 (数据库 + API + 基础 UI)
    → 测试通过，用户可以通过 UI 创建任务

切片 2: 列出任务 (查询 + API + UI)
    → 测试通过，用户可以看到自己的任务

切片 3: 编辑任务 (更新 + API + UI)
    → 测试通过，用户可以修改任务

切片 4: 删除任务 (删除 + API + UI + 确认)
    → 测试通过，完整的 CRUD 完成
```

每个切片交付可工作的端到端功能。

### 契约优先切片

当后端和前端需要并行开发时：

```
切片 0: 定义 API 契约 (类型、接口、OpenAPI 规范)
切片 1a: 后端按照契约实现 + API 测试
切片 1b: 前端按照匹配契约的模拟数据实现
切片 2: 集成并进行端到端测试
```

### 风险优先切片

先解决风险最高或最不确定的部分：

```
切片 1: 验证 WebSocket 连接可行 (最高风险)
切片 2: 在已验证的连接上构建实时任务更新
切片 3: 添加离线支持和重连
```

如果切片 1 失败，你在投入切片 2 和 3 之前就发现了问题。

## 实现规则

### 规则 0：简单优先

在编写任何代码之前，问自己："最简单的可行方案是什么？"

编写代码后，对照以下检查进行审查：
- 能用更少的代码实现吗？
- 这些抽象值得它们带来的复杂性吗？
- 一个高级工程师看到这个会说"为什么不直接..."吗？
- 我是在为假设的未来需求构建，还是当前任务？

```
简单性检查:
✗ 为了一个通知就搭建通用的 EventBus 和中间件管道
✓ 简单的函数调用

✗ 为两个相似组件使用抽象工厂模式
✓ 两个直接的组件加上共享工具函数

✗ 为三个表单搭建配置驱动的表单生成器
✓ 三个表单组件
```

三行相似的代码比过早的抽象更好。先实现简单、明显正确的版本。只有在测试证明正确性后才进行优化。

### 规则 0.5：范围纪律

只触及任务要求的部分。

不要：
- "顺便清理"你变更旁边的代码
- 重构你没有修改的文件中的 import
- 删除你不完全理解的注释
- 添加规范中没有的功能，因为它们"看起来有用"
- 对你只是在阅读的文件进行语法现代化

如果你注意到任务范围之外值得改进的东西，记下来——不要去修它：

```
注意到但不触及:
- src/utils/format.ts 有一个未使用的 import (与本任务无关)
- auth 中间件可以有更好的错误消息 (独立任务)
→ 需要我为这些创建任务吗？
```

### 规则 1：一次一件事

每个增量只改变一个逻辑上的事情。不要混合关注点：

**差：** 一个提交添加了新组件、重构了现有组件、还更新了构建配置。

**好：** 三个独立的提交——每个变更各一个。

### 规则 2：保持可编译

每个增量后，项目必须能构建，现有测试必须通过。不要在切片之间让代码库处于损坏状态。

### 规则 3：未完成功能使用特性开关

如果功能还没准备好给用户使用，但你需要合并增量：

```typescript
// Feature flag for work-in-progress
const ENABLE_TASK_SHARING = process.env.FEATURE_TASK_SHARING === 'true';

if (ENABLE_TASK_SHARING) {
  // New sharing UI
}
```

这让你可以将小的增量合并到主分支，而不暴露未完成的工作。

### 规则 4：安全默认值

新代码应默认为安全、保守的行为：

```typescript
// Safe: disabled by default, opt-in
export function createTask(data: TaskInput, options?: { notify?: boolean }) {
  const shouldNotify = options?.notify ?? false;
  // ...
}
```

### 规则 5：便于回滚

每个增量应能独立回滚：

- 添加性变更（新文件、新函数）容易回滚
- 对现有代码的修改应最小且聚焦
- 数据库迁移应有对应的回滚迁移
- 避免在一个提交中删除某样东西并在同一个提交中替换它——将它们分开

## 与 Agent 协作

当指导 Agent 进行增量实现时：

```
"让我们实现计划中的任务 3。

先只做数据库 schema 变更和 API 端点。
还不要碰 UI——我们会在下一个增量中做。

实现后，运行 `npm test` 和 `npm run build` 验证
没有破坏任何东西。"
```

对每个增量，明确说明什么在范围内，什么不在范围内。

## 增量检查清单

每个增量后，验证：

- [ ] 变更只做一件事并且完整地完成了
- [ ] 所有现有测试仍然通过 (`npm test`)
- [ ] 构建成功 (`npm run build`)
- [ ] 类型检查通过 (`npx tsc --noEmit`)
- [ ] 代码检查通过 (`npm run lint`)
- [ ] 新功能按预期工作
- [ ] 变更已用描述性消息提交

## 常见合理化借口

| 合理化借口 | 现实 |
|---|---|
| "最后再一起测" | Bug 会叠加。切片 1 中的 Bug 会导致切片 2-5 都出错。测试每个切片。 |
| "一次全做更快" | *感觉*更快，直到出了问题却找不到 500 行变更中的哪一行导致的。 |
| "这些变更太小，不值得分开提交" | 小提交没有成本。大提交隐藏 Bug 并使回滚痛苦。 |
| "之后再加特性开关" | 如果功能未完成，就不应对用户可见。现在就加。 |
| "这个重构足够小，可以一起包含" | 重构和功能混在一起让两者都更难审查和调试。分开它们。 |

## 危险信号

- 写了超过 100 行代码还没运行测试
- 一个增量中包含多个不相关的变更
- "让我顺便加上这个"的范围扩展
- 为了加速而跳过测试/验证步骤
- 增量之间构建或测试失败
- 大量未提交的变更在积累
- 在第三个用例需要之前就构建抽象
- "既然我在这里"就触及任务范围外的文件
- 为一次性操作创建新的工具文件

## 验证

完成一个任务的所有增量后：

- [ ] 每个增量都单独测试并提交了
- [ ] 完整的测试套件通过
- [ ] 构建是干净的
- [ ] 功能按规范端到端工作
- [ ] 没有未提交的变更残留
