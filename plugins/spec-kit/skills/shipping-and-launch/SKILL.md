# 发布与上线

## 概述

自信地发布。目标不仅仅是部署——而是安全地部署，确保监控就位、回滚方案就绪，并清楚地了解成功的标准。每次发布都应该是可逆的、可观测的和渐进的。

## 何时使用

- 首次将功能部署到生产环境
- 向用户发布重大变更
- 迁移数据或基础设施
- 开启 Beta 或抢先体验计划
- 任何有风险的部署（其实每次都有风险）

## 上线前检查清单

### 代码质量

- [ ] 所有测试通过（单元测试、集成测试、端到端测试）
- [ ] 构建成功且无警告
- [ ] Lint 和类型检查通过
- [ ] 代码已审查并批准
- [ ] 没有应在上线前解决的 TODO 注释
- [ ] 生产代码中没有 `console.log` 调试语句
- [ ] 错误处理覆盖了预期的故障模式

### 安全

- [ ] 代码或版本控制中没有密钥
- [ ] `npm audit` 没有 critical 或 high 级别漏洞
- [ ] 所有面向用户的端点都有输入验证
- [ ] 认证和授权检查已就位
- [ ] 安全响应头已配置（CSP、HSTS 等）
- [ ] 认证端点已启用速率限制
- [ ] CORS 配置为指定来源（非通配符）

### 性能

- [ ] Core Web Vitals（核心网页指标）在"良好"阈值内
- [ ] 关键路径中没有 N+1 查询
- [ ] 图片已优化（压缩、响应式尺寸、懒加载）
- [ ] 包体积在预算范围内
- [ ] 数据库查询有适当的索引
- [ ] 静态资源和重复查询已配置缓存

### 无障碍

- [ ] 所有交互元素支持键盘导航
- [ ] 屏幕阅读器能传达页面内容和结构
- [ ] 颜色对比度符合 WCAG 2.1 AA 标准（文本 4.5:1）
- [ ] 模态框和动态内容的焦点管理正确
- [ ] 错误消息描述清晰且与表单字段关联
- [ ] axe-core 或 Lighthouse 中没有无障碍警告

### 基础设施

- [ ] 生产环境变量已设置
- [ ] 数据库迁移已应用（或已准备就绪）
- [ ] DNS 和 SSL 已配置
- [ ] CDN 已配置用于静态资源
- [ ] 日志和错误上报已配置
- [ ] 健康检查端点存在且正常响应

### 文档

- [ ] README 已更新，包含所有新的配置要求
- [ ] API 文档是最新的
- [ ] 架构决策已编写 ADR（架构决策记录）
- [ ] 变更日志已更新
- [ ] 面向用户的文档已更新（如适用）

## 功能开关策略

使用功能开关（Feature Flag）将部署与发布解耦：

```typescript
// 功能开关检查
const flags = await getFeatureFlags(userId);

if (flags.taskSharing) {
  // 新功能：任务共享
  return <TaskSharingPanel task={task} />;
}

// 默认：保持现有行为
return null;
```

**功能开关生命周期：**

```
1. 部署时关闭开关         → 代码在生产环境但未激活
2. 对团队/Beta 用户启用   → 在生产环境中进行内部测试
3. 灰度发布               → 5% → 25% → 50% → 100% 的用户
4. 每个阶段监控           → 关注错误率、性能、用户反馈
5. 清理                   → 全量发布后移除开关和无用代码路径
```

**规则：**
- 每个功能开关都有负责人和过期日期
- 全量发布后 2 周内清理开关
- 不要嵌套功能开关（会产生指数级组合）
- 在 CI 中测试开关的两种状态（开和关）

## 分阶段发布

### 发布顺序

```
1. 部署到预发环境
   └── 在预发环境运行完整测试套件
   └── 关键流程手动冒烟测试

2. 部署到生产环境（功能开关关闭）
   └── 验证部署成功（健康检查）
   └── 检查错误监控（无新错误）

3. 对团队启用（内部用户开启开关）
   └── 团队在生产环境使用该功能
   └── 24 小时监控窗口

4. 金丝雀发布（5% 用户开启开关）
   └── 监控错误率、延迟、用户行为
   └── 对比指标：金丝雀 vs 基线
   └── 24-48 小时监控窗口
   └── 仅在所有阈值通过时推进（见下表）

5. 逐步扩大（25% -> 50% -> 100%）
   └── 每一步进行相同监控
   └── 任何时候都可以回退到上一个百分比

6. 全量发布（所有用户开启开关）
   └── 监控 1 周
   └── 清理功能开关
```

### 发布决策阈值

使用以下阈值决定在每个阶段是推进、暂停还是回滚：

| 指标 | 推进（绿灯） | 暂停并调查（黄灯） | 回滚（红灯） |
|--------|-----------------|-------------------------------|-----------------|
| 错误率 | 在基线 10% 以内 | 高于基线 10-100% | 高于基线 2 倍以上 |
| P95 延迟 | 在基线 20% 以内 | 高于基线 20-50% | 高于基线 50% 以上 |
| 客户端 JS 错误 | 无新错误类型 | 新错误影响 <0.1% 的会话 | 新错误影响 >0.1% 的会话 |
| 业务指标 | 中性或正向 | 下降 <5%（可能是噪声） | 下降 >5% |

### 何时回滚

出现以下情况时立即回滚：
- 错误率增加超过基线 2 倍
- P95 延迟增加超过 50%
- 用户报告问题激增
- 检测到数据完整性问题
- 发现安全漏洞

## 监控与可观测性

### 监控什么

```
应用指标：
├── 错误率（总体和按端点）
├── 响应时间（p50、p95、p99）
├── 请求量
├── 活跃用户数
└── 关键业务指标（转化率、参与度）

基础设施指标：
├── CPU 和内存利用率
├── 数据库连接池使用率
├── 磁盘空间
├── 网络延迟
└── 队列深度（如适用）

客户端指标：
├── Core Web Vitals（LCP、INP、CLS）
├── JavaScript 错误
├── 客户端视角的 API 错误率
└── 页面加载时间
```

### 错误上报

```typescript
// 设置带错误上报的错误边界
class ErrorBoundary extends React.Component {
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    // 上报到错误追踪服务
    reportError(error, {
      componentStack: info.componentStack,
      userId: getCurrentUser()?.id,
      page: window.location.pathname,
    });
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback onRetry={() => this.setState({ hasError: false })} />;
    }
    return this.props.children;
  }
}

// 服务端错误上报
app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  reportError(err, {
    method: req.method,
    url: req.url,
    userId: req.user?.id,
  });

  // 不向用户暴露内部细节
  res.status(500).json({
    error: { code: 'INTERNAL_ERROR', message: 'Something went wrong' },
  });
});
```

### 上线后验证

上线后第一个小时内：

```
1. 检查健康检查端点返回 200
2. 检查错误监控仪表板（无新错误类型）
3. 检查延迟仪表板（无退化）
4. 手动测试关键用户流程
5. 验证日志正在流转且可读
6. 确认回滚机制可用（尽可能进行演练）
```

## 回滚策略

每次部署前都需要一个回滚方案：

```markdown
## [功能/版本] 的回滚方案

### 触发条件
- 错误率 > 基线 2 倍
- P95 延迟 > [X]ms
- 用户报告 [特定问题]

### 回滚步骤
1. 禁用功能开关（如适用）
   或
1. 部署上一版本：`git revert <commit> && git push`
2. 验证回滚：健康检查、错误监控
3. 通知：告知团队已回滚

### 数据库注意事项
- 迁移 [X] 有回滚脚本：`npx prisma migrate rollback`
- 新功能插入的数据：[保留 / 清理]

### 回滚耗时
- 功能开关：< 1 分钟
- 重新部署上一版本：< 5 分钟
- 数据库回滚：< 15 分钟
```

## 常见自我合理化

| 合理化借口 | 现实 |
|---|---|
| "在预发环境可以，生产环境也没问题" | 生产环境有不同的数据、流量模式和边界情况。部署后要监控。 |
| "这个不需要功能开关" | 每个功能都应该有一个紧急关闭开关。即使"简单"的改动也可能出问题。 |
| "监控是额外开销" | 没有监控意味着你从用户投诉而非仪表板发现问题。 |
| "我们以后再加监控" | 上线前就加。看不到的东西无法调试。 |
| "回滚就是承认失败" | 回滚是负责任的工程实践。发布一个有缺陷的功能才是真正的失败。 |

## 危险信号

- 没有回滚方案就部署
- 生产环境没有监控或错误上报
- 大爆炸式发布（一次性全部上线，没有预发验证）
- 功能开关没有过期日期或负责人
- 部署后第一个小时没人监控
- 生产环境配置靠记忆而非代码
- "周五下午了，发布吧"

## 验证

部署前：

- [ ] 上线前检查清单已完成（所有项目通过）
- [ ] 功能开关已配置（如适用）
- [ ] 回滚方案已记录
- [ ] 监控仪表板已搭建
- [ ] 团队已收到部署通知

部署后：

- [ ] 健康检查返回 200
- [ ] 错误率正常
- [ ] 延迟正常
- [ ] 关键用户流程正常
- [ ] 日志正在流转
- [ ] 回滚已测试或确认可用
